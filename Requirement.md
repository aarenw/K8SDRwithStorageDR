## 2. 执行摘要

本项目面向制造业工厂环境，旨在利用 DELL PowerStore Metro Mirror 存储复制技术，结合双数据中心各自独立的 OpenShift 容器平台，构建一套覆盖虚拟机与容器应用的灾难恢复（DR）解决方案。当主数据中心（DC1）发生故障时，能够在备份数据中心（DC2）内以 RTO ≤ 1 小时的目标完成业务恢复，同时满足 RPO = 0 的数据零丢失要求。

---

## 3. 业务背景与现状

### 3.1 行业背景

客户为制造业工厂，生产环境对 IT 系统的连续性高度依赖。一旦核心业务系统中断，将直接影响生产线运作，造成重大经济损失。

### 3.2 现状描述

- IT 整体规模适中（中小型），但对**业务连续性要求极高**。
- 现有应用系统要求在发生故障后**能够快速恢复**，不允许长时间中断。
- 业务系统以虚拟机为主要部署形态，同时存在少量容器化应用。
- 目前已具备双数据中心基础设施，具有实施 DR 方案的硬件基础。

---

## 4. 技术环境

### 4.1 基础设施拓扑

| 组件           | DC1（主数据中心）         | DC2（备份数据中心）       |
| -------------- | ------------------------- | ------------------------- |
| 角色           | 生产主站点                | 灾备站点                  |
| 容器平台       | OpenShift 集群（独立）    | OpenShift 集群（独立）    |
| 虚拟化层       | OpenShift Virtualization  | OpenShift Virtualization  |
| 共享存储       | DELL PowerStore           | DELL PowerStore           |

### 4.2 存储复制

- 存储产品：**DELL PowerStore**
- 复制模式：**Metro Mirror**（同步镜像）
- 复制粒度：Volume（块存储 LUN 级别）
- RPO 目标：**0**（同步复制，无数据丢失）

### 4.3 工作负载规模

| 类型         | 数量     | 备注                           |
| ------------ | -------- | ------------------------------ |
| 虚拟机（VM） | 200+     | 主要业务载体                   |
| 容器应用     | 少量     | 部署于 OpenShift 容器平台之上  |

---

## 5. 项目目标

### 5.1 主要目标

在主数据中心 DC1 发生灾难性故障时，能够借助 DC2 的 OpenShift 集群及存储镜像数据，**快速恢复全部业务系统的正常运行**。

### 5.2 可量化成功指标

| 指标 | 目标值       | 说明                                         |
| ---- | ------------ | -------------------------------------------- |
| RPO  | **0**        | 基于 Metro Mirror 同步复制，故障前数据零丢失 |
| RTO  | **≤ 1 小时** | 从故障确认到所有关键业务系统恢复可用         |

---

## 6. 需求规格

### 6.1 功能需求

| 编号  | 需求描述                                                                                  | 优先级 |
| ----- | ----------------------------------------------------------------------------------------- | ------ |
| FR-01 | 系统应支持将 DC1 上所有 200+ 台虚拟机在 DC2 完成 Failover 并启动                         | 高     |
| FR-02 | 系统应支持将 DC1 上少量容器应用在 DC2 完成 Failover 并恢复运行                            | 高     |
| FR-03 | Failover 时，应提供机制修改 VM 的 IP 地址以适配 DC2 网络（VLAN 统一，但 IP 需变更）       | 高     |
| FR-04 | Failover 流程应覆盖对 SAN 存储访问权限的切换，防止 DC1 与 DC2 同时挂载同一 LUN            | 高     |
| FR-05 | 应提供 Failover 操作手册或自动化脚本，支持运维人员在规定时间内完成切换                    | 中     |
| FR-06 | 应支持 Failback（从 DC2 回切到 DC1）流程                                                  | 中     |

### 6.2 非功能需求

| 编号   | 类别     | 需求描述                                                                       | 目标值       |
| ------ | -------- | ------------------------------------------------------------------------------ | ------------ |
| NFR-01 | 可用性   | 主数据中心完全故障后，业务系统恢复时间                                         | ≤ 1 小时     |
| NFR-02 | 数据完整性 | 基于 Metro Mirror 同步复制，Failover 后数据与故障前一致                      | RPO = 0      |
| NFR-03 | 扩展性   | 方案需支持 200+ 台 VM 的并发 Failover 操作                                    | ≥ 200 台 VM  |
| NFR-04 | 可操作性 | Failover 操作步骤应有完整文档，支持运维人员独立执行，无需开发介入              | —            |
| NFR-05 | 安全性   | 同一时间只允许一个数据中心对存储 LUN 具有读写权限，防止数据损坏               | —            |

## 7. 假设与约束

### 7.1 假设条件

| 编号  | 假设内容                                                                                     |
| ----- | -------------------------------------------------------------------------------------------- |
| AS-01 | DC1 与 DC2 的网络位于**同一 VLAN**，Failover 后 VM 的网络 IP 地址需要手动或自动修改         |
| AS-02 | 存储 Mirror 后，两个数据中心的 **Volume LUN ID 保持不变**，无需重新映射                     |
| AS-03 | 应用的 **DNS 名称在 Failover 前后保持不变**，通过 DNS 指向变更实现流量切换（如适用）         |

### 7.2 约束条件

| 编号  | 约束内容                                                                     |
| ----- | ---------------------------------------------------------------------------- |
| CO-01 | 方案基于现有双数据中心基础设施，不涉及新数据中心建设                         |
| CO-02 | 存储复制技术限定为 **DELL PowerStore Metro Mirror**                          |
| CO-03 | 容器平台限定为 **Red Hat OpenShift**，虚拟化层限定为 **OpenShift Virtualization** |

## 8. 风险与缓解措施

| 编号  | 风险描述                                                                                     | 影响 | 概率 | 缓解措施                                                                                     |
| ----- | -------------------------------------------------------------------------------------------- | ---- | ---- | -------------------------------------------------------------------------------------------- |
| R-01  | **脑裂（Split-Brain）**：DC1 与 DC2 同时挂载并写入同一 SAN LUN，导致数据损坏                 | 极高 | 中   | VM 不允许在 DC2 自动启动；可通过 **SAN 交换机 Zone 配置**限制各数据中心对存储的访问权限       |
| R-02  | **镜像仓库依赖瓶颈**：Failover 时大量 VM 和容器同时拉取镜像，导致 Image Repository 过载      | 高   | 中   | 在 DC2 部署本地镜像仓库；Failover 演练中评估并优化镜像预拉取策略                             |
| R-03  | **控制平面响应瓶颈**：OpenShift 控制平面（API Server、etcd）在高并发 Failover 时响应缓慢      | 高   | 中   | Failover 操作分批次执行，避免并发峰值；监控控制平面负载                                       |
| R-04  | **RTO 超标**：Failover 步骤串行执行导致恢复时间超过 1 小时                                    | 高   | 中   | 识别可并行执行的操作（如多批次 VM 并发启动），并在操作手册中明确并行化策略                    |
| R-05  | **VM IP 修改遗漏**：Failover 后 VM IP 未及时更新，导致网络不通                                | 中   | 中   | 建立 VM IP 变更清单，自动化 IP 修改流程                                                       |

## 9. 依赖与前提条件

| 编号  | 依赖项                                                               | 责任方       |
| ----- | -------------------------------------------------------------------- | ------------ |
| D-01  | DELL PowerStore Metro Mirror 已在 DC1 与 DC2 之间正常同步            | 存储团队     |
| D-02  | DC2 OpenShift 集群已部署完毕且处于健康状态                           | 平台团队     |
| D-03  | SAN 交换机具备 Zone 隔离能力，且运维团队有权限操作                   | 网络/存储团队 |
| D-04  | DC2 具备本地镜像仓库或可访问的镜像源，保障 Failover 时容器正常启动   | 平台团队     |
| D-05  | DNS 解析服务在 DC2 侧可用，支持 Failover 后的名称解析                | 网络团队     |

## 10. 验收标准

| 编号  | 验收项                                                                                   | 通过条件                                     |
| ----- | ---------------------------------------------------------------------------------------- | -------------------------------------------- |
| AC-01 | **RTO 达标**：在模拟 DC1 完全故障的 Failover 演练中，所有关键业务在 DC2 恢复正常运行    | 恢复时间 ≤ 1 小时                            |
| AC-02 | **数据完整性**：Failover 后，DC2 存储数据与 DC1 故障前最后一次写入数据完全一致          | 数据校验无差异，RPO = 0                      |
| AC-03 | **无脑裂发生**：Failover 全程，DC1 与 DC2 不同时对同一 LUN 进行写操作                   | 演练期间无双写告警或数据损坏                 |
| AC-04 | **VM 网络正常**：Failover 后所有 VM 完成 IP 修改，应用层网络连通                        | VM IP 修改完成率 100%                        |
| AC-05 | **并行化验证**：Failover 操作手册中标注的可并行步骤，在实际演练中可同步执行且互不干扰   | 并行执行期间无冲突或错误                     |
| AC-06 | **Failback 可行**：验证从 DC2 回切至 DC1 的流程可执行                                   | Failback 流程文档完整，演练通过              |

## 11. 术语表

| 术语                    | 说明                                                                                 |
| ----------------------- | ------------------------------------------------------------------------------------ |
| DR（Disaster Recovery） | 灾难恢复，当主站点发生故障时，在备用站点恢复业务的能力                               |
| RPO（Recovery Point Objective） | 恢复点目标，即故障发生时最多可接受丢失的数据量（以时间衡量）               |
| RTO（Recovery Time Objective）  | 恢复时间目标，即从故障发生到业务恢复正常所允许的最长时间                   |
| Metro Mirror            | 同城/同步镜像复制技术，两端存储数据实时一致，RPO = 0                                 |
| Failover                | 故障切换，将业务从主站点切换到备用站点的操作过程                                     |
| Failback                | 故障回切，将业务从备用站点切换回主站点的操作过程                                     |
| DC1 / DC2               | 主数据中心（Data Center 1）/ 备份数据中心（Data Center 2）                           |
| LUN（Logical Unit Number） | 逻辑单元号，SAN 存储中对外提供的逻辑磁盘单元                                     |
| SAN（Storage Area Network） | 存储区域网络，提供块级别存储访问的专用高速网络                                   |
| OpenShift               | Red Hat 企业级 Kubernetes 容器平台                                                   |
| OpenShift Virtualization | OpenShift 上运行虚拟机的能力，基于 KubeVirt 实现                                   |
| Zone                    | SAN 交换机上的访问控制分组，限制哪些主机可以访问哪些存储端口                         |
